# 你的 OpenClaw 真的在受控运行吗？

**Author:** 徐可甲 (Alibaba Cloud)
**Published:** 2026-03-02
**Source:** Alibaba Cloud WeChat official account (阿里云云原生)
**URL:** https://mp.weixin.qq.com/s?__biz=MzUzNzYxNjAzMg==&mid=2247582344

## Summary

A practical guide to building a full-stack observability system for [[3 - Areas/AIOps/wiki/entities/OpenClaw]] AI Agents using [[3 - Areas/AIOps/wiki/entities/Alibaba Cloud SLS|Alibaba Cloud SLS (Simple Log Service)]]. The article frames the core question as: *"Is your Agent truly running under control?"* — and answers it by establishing three coordinated data pipelines that together constitute [[3 - Areas/AIOps/wiki/concepts/AI Agent Observability]].

The key argument: runtime defences (tool policies, loop detectors) are the "city wall"; observability is the "watchtower" — only continuous observation can catch what the wall misses.

---

## Key Takeaways

### The Four Questions of Controlled Operation
An Agent is only "controlled" if you can answer all four:
1. **Who** is triggering the calls?
2. **How much** is it costing?
3. **What operations** were performed (especially high-risk tools)?
4. **Is the behaviour traceable and auditable?**

### AI Agent Security Risk Taxonomy

| Risk | Scenario | Consequence |
|---|---|---|
| Tool/Skill abuse | Agent induced to run `exec` with malicious command | System intrusion |
| Data exfiltration | Agent reads sensitive file via tool, outputs to conversation | Privacy breach |
| Cost runaway | Agent enters loop, continuously consuming tokens | Bill explosion |
| Prompt injection | User embeds instructions overriding System Prompt | Behaviour hijack |
| Session hijack | Multi-turn conversation gradually steers Agent off course | Auth bypass |

### Three-Pillar Mapping for AI Agents

| Pillar | OpenClaw Data Source | Core Questions Answered |
|---|---|---|
| Session Logs | `~/.openclaw/agents/<id>/sessions/*.jsonl` | What did the Agent do? Which tools? What parameters? What cost? |
| App Logs | `/tmp/openclaw/openclaw-YYYY-MM-DD.log` | What system errors occurred? Gateway issues? Queue backlog? |
| Metrics (OTEL) | `diagnostics-otel` plugin OTLP output | Current cost? Latency normal? Any stuck sessions? |
| Traces (OTEL) | `diagnostics-otel` plugin OTLP output | Full request chain: receive → response, with timing |

> [!tip] Session logs are a 4th signal unique to AI Agents
> Classical observability has three pillars (logs/metrics/traces). AI Agents introduce **session audit logs** — structured JSONL recording every conversation turn, tool call, parameter, and token cost. This is the primary forensic data source for security audits.

### Session Log Format
Each session produces a `.jsonl` file. Each line is typed:
- `message` with `role: user` — user input
- `message` with `role: assistant` + `toolCall` content — LLM decision with tool invocations
- `message` with `role: toolResult` — tool execution output
- `message` with `role: assistant` + `stopReason: stop` — final response with `usage.cost`

Key fields: `message.provider`, `message.model`, `message.usage.totalTokens`, `message.usage.cost.total`, `message.stopReason`

### High-Risk Tool Monitoring
Two categories of tools to monitor in session logs:
- **Gateway HTTP blocked by default**: `sessions_spawn`, `sessions_send`, `cron`, `gateway`, `whatsapp_login`
- **ACP (Automation Control Plane) explicit-approval required**: `exec`, `spawn`, `shell`, `fs_write`, `fs_delete`, `fs_move`, `apply_patch`

### OTEL Metrics Exported by OpenClaw
- **Cost & usage**: `openclaw.tokens`, `openclaw.cost.usd`, `openclaw.run.duration_ms`, `openclaw.context.tokens`
- **Webhook**: `openclaw.webhook.received`, `openclaw.webhook.error`, `openclaw.webhook.duration_ms`
- **Queue**: `openclaw.message.queued/processed/duration_ms`, `openclaw.queue.depth/wait_ms`
- **Session health**: `openclaw.session.state`, `openclaw.session.stuck`, `openclaw.session.stuck_age_ms`

### Multi-Source Correlation Workflow
The three pipelines answer different questions and must be used together:
- **OTEL alone**: know cost is spiking, but not who or what caused it
- **Session logs alone**: can audit behaviour but can't sense overall system state
- **App logs alone**: can see system errors but not Agent business behaviour
