# 别再被 AI 反噬：DevOps 误用清单，守住你的技术栈安全边界

## Summary

A practitioner retrospective on five common patterns of AI misuse in DevOps, triggered by a real incident where AI-generated Terraform code exposed a production PostgreSQL RDS instance to the public internet (0.0.0.0/0 security group rule), costing $50K in emergency response. The article defines a framework for safely using AI as a "coding accelerator" without ceding engineering judgment.

The core framing: treat AI-generated infrastructure code like code from a brilliant but reckless junior engineer — always review, never blindly deploy.

---

## Key Takeaways

### The Root Problem: AI Training Data Bias

AI models are trained on millions of GitHub/StackOverflow examples where the priority is **"works"**, not **"secure"**. The "happy path" problem: AI generates configurations that run immediately in dev, but are unsuitable for production without explicit security constraints.

### The 5 Anti-Patterns

**Anti-pattern 1 — AI-generated infra ≠ secure infra**
Default AI output includes overly permissive security groups (0.0.0.0/0), open SSH ports. Fix: explicitly specify "production-ready, least-privilege" in every infrastructure prompt.

**Anti-pattern 2 — Writing prompts like a developer, not a DevOps engineer**
Vague prompts ("write a Python GitHub Actions workflow") yield basic CI without secrets management, rollback, or error handling. DevOps-grade prompts specify deployment strategy (blue/green), auth (OIDC not stored secrets), security scanning (Snyk), integration tests, health-check rollback, artifact retention, and notifications.

**Anti-pattern 3 — False confidence from clean-looking AI output**
AI output has no TODO comments or ambiguous variable names — it looks authoritative. A K8s Deployment with no health checks, `latest` image tag, insufficient resource limits, no security context, and no namespace constraint looks fine until production.

**Anti-pattern 4 — Not making AI think like an SRE**
Before generating any infra code, set a system prompt establishing the AI as a Fortune 500 SRE responsible for 99.99% uptime, zero-trust security, SOC2/PCI-DSS compliance, and cost optimization. The resulting Terraform is structurally different — multi-AZ, encrypted, versioned, with CloudWatch alarms.

**Anti-pattern 5 — Blindly trusting AI-generated YAML**
K8s YAML anti-patterns: secrets in plain-text env vars, `latest` image tags, missing resource limits (one memory leak kills the node), container running as root, no health checks.

### The "AI Audits AI" Technique

> [!tip] Key insight
> After generating infrastructure code, immediately follow up with a security-audit prompt asking the AI to act as a paranoid Senior Security Engineer reviewing its own output for PCI-compliant payment workloads. In one real case this surfaced 7 previously missed issues (2× Critical, 2× High, 2× Medium, 1× Low).

The audit prompt template:
```
You are a Senior Security Engineer conducting a security audit of the [RESOURCE TYPE] you just generated. Assume worst-case: PCI-compliant data, shared cluster, sophisticated attackers. Provide: severity rating, specific line, exact fix, why it matters in production. Be paranoid.
```

This technique — using AI to validate AI — becomes a standard part of the workflow: generate → audit → fix → validate with automated tools.

### The 60/40 Rule

**AI handles 60% of the typing; 100% of the thinking stays with the engineer.**

AI is appropriate for:
- Generating boilerplate code
- Exploring different implementation approaches
- Finding errors you might have missed
- Researching best practices for unfamiliar technologies

Engineers must still:
- Understand what the generated code does
- Verify it meets security requirements
- Ensure it fits existing architecture
- Test thoroughly in context
- Make architectural decisions based on business reality

Real failure: an AI-generated autoscaling config that was technically correct but added $10K/month in costs because it didn't understand the actual traffic patterns.

### Validation Pipeline

Automated gates every AI-generated IaC must pass before merge:

| Tool | Purpose |
|---|---|
| `tfsec` | Terraform security scanning |
| `Checkov` | Multi-language IaC security |
| `Snyk` | Code and container vulnerability scanning |
| `Trivy` | Container and filesystem vuln scanning |
| `SemGrep` | Custom security rules / SAST |
| `kube-score` | Kubernetes object analysis |
| `kubeval` | YAML schema validation |
| `OPA Gatekeeper` | Policy enforcement |
| `Falco` | Runtime security monitoring |
| `Infracost` | Cost estimation before deployment |

GitHub Actions safety gate catches common AI mistakes automatically: 0.0.0.0/0 CIDR blocks, hardcoded secrets, `latest` image tags, K8s Deployments without resource limits.
